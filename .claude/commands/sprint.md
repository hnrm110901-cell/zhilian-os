## 流程

1. **加载上下文**：读取 `tasks/lessons.md` 防止重蹈覆辙，再读 `ARCHITECTURE.md` 确认Ontology约束
2. **检查环境**：确认当前分支和未提交变更（`git status` + `git diff --stat`）
3. **理解任务**：分析 $ARGUMENTS 描述的任务范围，定位涉及的模块和文件
4. **制定计划**：输出变更蓝图（拟变更文件 + 每文件变更意图 + 风险点），**不要立即写代码，等待确认**
5. **执行开发**（计划确认后）：
   - 先写单元测试（pytest-asyncio 用于异步代码）
   - 再实现功能代码
   - 最后写集成测试
6. **数据库检查**：如涉及模型变更，运行 `alembic check` 确认无数据库漂移
7. **验证**：在对应目录运行 `pytest` 确认全部通过
8. **提交**：生成规范的 git commit message（中文描述 + 英文标签），格式：`feat(模块): 中文描述`

## 质量关卡

- 所有金额输出包含 `¥` 字段（Rule 6）
- 推送/建议包含：建议动作 + 预期¥影响 + 置信度（Rule 7）
- SQL 使用参数化查询，无 f-string（L010/L011）
