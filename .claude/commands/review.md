对当前分支的变更执行智链OS代码审查：$ARGUMENTS

## 审查流程

1. **获取变更范围**：`git diff master...HEAD --stat` 列出所有变更文件
2. **逐文件审查**：读取每个变更文件，对照以下检查清单
3. **输出审查报告**：按严重程度分级（BLOCKER / WARNING / SUGGESTION）

## 检查清单

### 安全（BLOCKER 级）
- [ ] SQL 查询是否使用参数化绑定（无 f-string / 字符串拼接）（L010/L011）
- [ ] INTERVAL 是否用 `:n * INTERVAL '1 day'` 形式（L010）
- [ ] JSON 构造是否用 `json.dumps()`（L013）
- [ ] `except Exception: pass` 是否有日志记录（L014）
- [ ] 外部输入是否经过验证

### 正确性（BLOCKER 级）
- [ ] 除法运算是否显式判断分母 > 0（L006）
- [ ] 分页接口 `total` 是否为真实总量（L015）
- [ ] 金额单位是否正确（DB存分，API返回元）
- [ ] 条件分支是否每个都可达（L007）
- [ ] 方法重命名后测试是否同步更新（L012）
- [ ] 前后端类型定义是否一致（L018）

### 产品规则
- [ ] 涉及成本/收入的输出是否包含 `¥金额` 字段（Rule 6）
- [ ] 推送内容是否包含：建议动作 + 预期¥影响 + 置信度 + 操作入口（Rule 7）
- [ ] 是否在10个MVP功能范围内（Rule 8）
- [ ] 查询类功能是否有离线降级方案（Rule 10）

### 工程规范
- [ ] 命名是否符合规范（PascalCase类 / snake_case函数 / camelCase CSS）
- [ ] 是否有 TODO/FIXME 残留
- [ ] 测试覆盖是否充分（每个公开方法至少1个测试）
- [ ] 新增模型是否已在 `__init__.py` 注册

## 输出格式

```
## 审查结果：{通过/需修改}

### BLOCKER（必须修复）
1. `file:line` — 问题描述 — 修复建议

### WARNING（建议修复）
1. `file:line` — 问题描述

### SUGGESTION（可选优化）
1. 描述
```
