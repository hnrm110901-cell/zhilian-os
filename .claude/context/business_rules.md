## 核心业务概念

### 多租户隔离
- **brand_id**：品牌/连锁（如"徐记海鲜"）
- **store_id**：单店（如"徐记海鲜长沙店"）
- 所有查询必须带 store_id 过滤
- 跨店分析需要 brand_id 过滤

### 金额体系
- **数据库**：存分（fen），BigInteger（部分历史表用 Numeric 存元）
- **API 响应**：返回元（yuan），保留2位小数
- **展示**：加 `¥` 前缀（Rule 6）
- **转换**：`amount_yuan = amount_fen / 100`

### 餐厅时段
| 时段 | 时间 | 英文 |
|------|------|------|
| 早餐 | 06:00-10:00 | breakfast |
| 午餐 | 10:00-14:00 | lunch |
| 下午茶 | 14:00-17:00 | afternoon |
| 晚餐 | 17:00-21:00 | dinner |
| 夜宵 | 21:00-02:00 | late_night |

## Agent 决策规则

### 排班决策（Schedule Agent）
1. **需求驱动**：客流预测 → 各岗位人力需求 → 排班建议
2. **三档降级**：rule_based（规则）→ statistical（统计）→ weighted（加权历史）
3. **节假日系数**：CN 法定假日 × 1.2-1.5，周末 × 1.1-1.3
4. **岗位需求公式**：`前厅 = ceil(预测客流 / 服务能力系数)`
5. **输出必须包含**：排班建议 + ¥人力成本预估 + 置信度

### 库存决策（Inventory Agent）
1. **预警规则**：`current_stock <= min_stock` 触发补货建议
2. **需求预测**：历史销量 × 安全系数 × 季节因子
3. **采购建议**：供应商最优价 × 需求量 = ¥采购金额
4. **损耗追踪**：WasteEvent 记录 → 损耗率分析

### 人力成本决策（Workforce）
1. **成本率阈值**：
   - ≤ 25%：健康（绿色）
   - 25%-30%：预警（黄色）
   - 30%-35%：警告（橙色）
   - ≥ 35%：危险（红色）
2. **流失风险评分**（0-1）：
   - < 0.3：low（绿色）
   - 0.3-0.5：medium（黄色）
   - 0.5-0.7：high（橙色）
   - ≥ 0.7：critical（红色，触发企微预警）
3. **每日推送**：07:00 企微推送「今日人力建议」

### 菜品决策（Dish R&D Agent）
1. **毛利率目标**：≥ 60% 为健康
2. **排名规则**：综合评分 = 销量权重 × 0.4 + 毛利权重 × 0.3 + 评分权重 × 0.3
3. **淘汰建议**：连续30天销量为0 + 毛利率 < 40%

### 宴会决策（Banquet Agent）
1. **线索管线**：新线索 → 跟进 → 报价 → 签约 → 执行 → 复盘
2. **报价策略**：套餐基础价 + 场地费 + 服务费 + 酒水加成
3. **应收管理**：定金 → 中期款 → 尾款，逾期触发催收
4. **KPI**：成交率 / 客单价 / 应收回款率

## 推送规则（Rule 7 合规）

### 推送格式要求
每条推送必须包含：
1. **建议动作**：明确告诉用户该做什么
2. **预期¥影响**：量化收益/成本
3. **置信度**：推理链的可信度（高/中/低）
4. **一键操作入口**：按钮 callback 接口

### 推送频率限制
- 每日每店 ≤ 3 条关键推送
- 非紧急推送合并为晨报/晚报
- 紧急推送（库存断货/成本超标）即时发送

### 企微消息格式
```python
# 正确：使用 json.dumps 构造消息体
import json
payload = json.dumps({"text": {"content": message}})

# 错误：字符串拼接（L013）
payload = '{"text":"' + message + '"}'
```

## BFF 聚合规则

### 店长首屏（/api/v1/bff/sm/{store_id}）
聚合数据：
- 今日营收 / 客流 / 成本率
- AI 建议（最多3条，按优先级排序）
- 待办事项（未确认的排班建议/库存预警）
- 关键 KPI 趋势

### 总部首屏（/api/v1/bff/hq/{store_id}）
聚合数据：
- 多店营收对比排名
- 人力成本率排名
- 全局 AI 建议
- 异常门店预警

### 缓存策略
- Redis TTL：30秒
- 强制刷新：`?refresh=true`
- 子调用降级：失败返回 `null`，前端 `ZEmpty` 占位

## 离线降级规则（Rule 10）

| 场景 | 降级方案 |
|------|---------|
| Redis 不可用 | 直接查 DB（性能降低但功能正常） |
| DB 不可用 | 返回最近缓存数据 + 标记 `stale: true` |
| LLM API 不可用 | 使用规则引擎替代（rule_based 降级） |
| 向量DB 不可用 | 返回零向量，RAG 质量下降但不报错 |
| 外部 POS 不可用 | 使用最近同步数据 + 标记 `last_sync_at` |

## 案例采集规则（Rule 9）

每个 Sprint 结束时确认以下数据可被 `case_story_generator.py` 采集：
- 成本率变化（本 Sprint 前 vs 后）
- 节省¥金额（量化）
- 决策采纳数（店长确认了多少条建议）
- 使用时长（系统活跃度）
