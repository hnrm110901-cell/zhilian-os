# 屯象OS · 三端协同原则手册
> **版本** v1.0 · **状态** 草案待三端共识 · **日期** 2026-03-16
> **适用范围** Claude Code · Claude CLI · Codex
> **仓库** https://github.com/hnrm110901-cell/zhilian-os.git

---

## 前言：为什么需要这份文档

屯象OS 同时由 Claude Code、Claude CLI、Codex 三端并行驱动开发，腾讯服务器（42.194.229.21）承载部署，GitHub 作为唯一事实来源。

没有共同的原则，三端各有各的节奏，就会产生：代码风格冲突、分支管理混乱、重复造轮子、合并地狱、文档缺失。

本文档汲取了 **DataMap vibe coding 实验**（Claude AI 写出 95% 代码的开源项目）的第一手经验，结合屯象OS实际情况，形成三端共识的工作原则。

**这不是规定，是共识。只有三端都认可的条目，才写入正式原则。**

## 第一章：三端分工宪章

### 1.1 职责边界

| 工具 | 主战场 | 禁止越界 |
|------|--------|---------|
| **Claude Code** | 架构设计、核心系统模块、跨模块重构、复杂逻辑 | 不写一次性脚本、不碰 UI 样式细节 |
| **Claude CLI** | 自动化脚本、CI 辅助、文档生成、批量操作、测试批跑 | 不做架构决策、不直接改核心模块 |
| **Codex** | UI 组件、模块补全、单元测试生成、API 适配层 | 不修改系统内核路径、不改 CI 配置 |

### 1.2 目录主权

```
zhilian-os/
├── src/core/          ← Claude Code 主导（需 review 才能合并）
├── src/ui/            ← Codex 主导
├── src/api/           ← Codex + Claude Code 共同维护
├── scripts/           ← Claude CLI 主导
├── tests/             ← 三端共同维护
├── docs/              ← Claude CLI 负责生成，三端共同审阅
└── .github/workflows/ ← Claude CLI 主导，Claude Code 审核
```

### 1.3 人类开发者的角色

借鉴 DataMap 经验：**开发者是产品经理，不是打字员。**

- 描述目标和约束，不逐行指定实现
- 验证运行结果，不只看代码是否语法正确
- 把控架构方向，在三端产生分歧时做最终裁决
- 识别 AI 生成代码中的**逻辑错误**（语法正确但逻辑错误是最危险的盲区）

## 第二章：Git 工作流原则

### 2.1 分支命名规范

```
main              ← 生产环境，仅接受来自 develop 的 PR，保护分支
develop           ← 集成分支，三端的合并目标
feat/cc-<功能>    ← Claude Code 功能分支（cc = claude-code）
feat/cli-<功能>   ← Claude CLI 功能分支
feat/cx-<功能>    ← Codex 功能分支
fix/<问题描述>    ← Bug 修复（任意端均可）
refactor/<范围>   ← 重构分支（通常由 Claude Code 发起）
docs/<范围>       ← 文档更新（通常由 Claude CLI 生成）
```

### 2.2 小步提交原则（最高优先级原则）

> **DataMap 第一教训**：野心太大的改动经常让应用崩溃，导致痛苦的回滚和进度损失。

**强制规则**：
- 每完成一个**可独立运行**的最小功能单元，立即 commit
- 单次 commit 改动行数建议不超过 200 行（重构除外）
- 绝不攒功能——不允许"等这几个功能都做完再一起提交"
- commit 前必须本地验证可运行

**判断标准**：能回答"这个 commit 让屯象OS多了什么能力？"就是好的 commit。

### 2.3 Commit 消息规范（Conventional Commits）

```
<类型>(<范围>): <简短描述>

类型:
  feat     - 新功能
  fix      - Bug 修复
  refactor - 重构（不改变外部行为）
  docs     - 文档
  test     - 测试
  chore    - 构建/CI/依赖
  perf     - 性能优化

范围（对应目录主权）:
  kernel   - src/core/ 内核相关
  ui       - src/ui/
  api      - src/api/
  ci       - .github/workflows/
  scripts  - scripts/
  docs     - docs/

示例:
  feat(kernel): 添加内存分配器基础接口
  fix(ui): 修复任务栏在深色模式下的渲染异常
  chore(ci): 配置 develop 分支自动部署到 staging
  docs(api): Claude CLI 生成进程管理模块 API 文档
```

### 2.4 PR 合并规则

| 目标分支 | 要求 |
|---------|------|
| `develop` | CI 通过 + 至少 1 人（或另一端工具）review |
| `main` | CI 通过 + 人工 review + 手动触发部署确认 |

**禁止行为**：
- 禁止直接 `push` 到 `main` 或 `develop`
- 禁止 `--force push`（紧急情况须报备）
- 禁止无描述的 PR（必须说明做了什么、为什么这么做）

## 第三章：AI 辅助编码原则

### 3.1 自我审查步骤（不可省略）

> **DataMap 经验**：单次 AI 生成的方案很少是最优的。定期让 Claude "彻底审查你的代码"往往能发现被忽视的错误。

每个功能模块完成后，在提 PR 之前，**必须执行自审流程**：

```
步骤 1：让 Claude Code 审查自己生成的代码
  提示词模板："请彻底审查以上代码，重点检查：
  1. 逻辑正确性（不只是语法）
  2. 边界条件和错误处理
  3. 与现有模块的接口兼容性
  4. 是否有更简洁的实现方式"

步骤 2：如果审查反复发现新问题 → 暂停，升级为人工干预
步骤 3：自审通过后才能开 PR
```

### 3.2 逻辑错误优先于语法错误

> **DataMap 警示**：Claude 频繁在数据分析步骤中引入微妙的逻辑错误。这些错误更难被发现，因为 AI 生成的代码通常语法正确并能顺利执行，从而掩盖了底层的逻辑问题。

屯象OS验收标准：
- ✅ 代码可以运行 → 只是基础门槛
- ✅ 运行结果符合预期 → 才是真正的验收
- ✅ 边界情况处理正确 → 才能合入 develop

### 3.3 上下文投递策略

| 任务规模 | 工具选择 | 上下文投递方式 |
|---------|---------|--------------|
| 单文件小改动 | Claude CLI / Copilot | 直接传当前文件 |
| 单模块功能开发 | Claude Code | 传模块相关文件 + 接口定义 |
| 跨模块重构 | Claude Code | 手动上传所有相关模块代码片段 |
| 全仓库分析 | Claude.ai + UitHub 快照 | 将 GitHub URL 的 "github" 替换为 "uitHub" 获取全仓文本 |
| UI 组件补全 | Codex | 组件文件 + 设计规范文档 |

### 3.4 重构时机原则

```
阶段一（快速迭代期）：先跑通，允许技术债
  → 三端各自按功能分支快速推进，不过度追求完美

阶段二（模块稳定后）：专项重构 Sprint
  → 由 Claude Code 主导，批量拆解耦合，提取模块
  → DataMap 经验：Claude 的重构能力极强，通常一次成功

原则：不在功能开发期重构，不在重构期加新功能
```

## 第四章：部署与服务器原则

### 4.1 双环境隔离

```
腾讯服务器 42.194.229.21
├── /opt/zhilian-os/staging/   ← develop 分支自动同步（测试环境）
└── /opt/zhilian-os/prod/      ← main 分支手动触发（生产环境）
```

### 4.2 自动化部署流程

```
push 到 develop
  → GitHub Actions 触发
  → lint + 自动测试
  → SSH 推送 staging 环境
  → pm2 reload
  → 健康检查
  → 通知（Webhook/消息）

push 到 main（仅通过 PR）
  → 同上流程
  → 额外：人工确认后才执行 prod 部署
```

### 4.3 无服务器优先原则

借鉴 DataMap 的 WebAssembly 无服务器架构：

- 工具类模块、配置界面、监控面板 → 优先考虑**静态化部署**（GitHub Pages）
- 数据处理逻辑 → 优先考虑**客户端执行**（减少服务器压力）
- 服务器只承载必须在服务端的核心服务

好处：降低服务器成本、提升安全性（数据不上传）、自动获得 CDN 加速。

## 第五章：文档原则

### 5.1 文档即代码

- 文档随代码同步更新，不允许"等稳定了再写文档"
- 每个稳定模块，由 Claude CLI 自动生成初版文档（README + API 文档）
- 人工审阅后 commit，与代码在同一 PR 中

### 5.2 文档生成模板（Claude CLI 标准提示）

```bash
# 生成模块文档的标准提示模板
"请为 src/core/<模块名>/ 生成：
1. README.md（功能说明、安装、使用示例）
2. API.md（所有公开接口的签名、参数、返回值、示例）
3. CHANGELOG 条目（本次新增/变更内容）
语言：中文，代码示例：完整可运行"
```

### 5.3 跨端理解同步

每次重大架构决策后，在 `docs/decisions/` 目录下记录 ADR（架构决策记录）：

```markdown
# ADR-<编号>: <决策标题>

## 状态
提议 / 已接受 / 已废弃

## 背景
为什么需要做这个决策？

## 决策
我们决定做什么？

## 后果
这个决策带来什么影响？什么变得更容易/更困难？
```

## 第六章：质量门禁

### 6.1 合并前检查清单（每个 PR 必须通过）

```
代码质量
  □ 本地运行验证通过（不只是语法检查）
  □ 逻辑边界条件已测试
  □ AI 自审已执行（见第三章 3.1）

Git 规范
  □ commit 消息符合 Conventional Commits 格式
  □ 分支命名符合规范
  □ 单次 PR 改动范围合理（非大爆炸式提交）

文档
  □ 新增公开接口已有文档
  □ 破坏性变更已在 PR 描述中说明

测试
  □ 新功能有对应测试用例
  □ 现有测试全部通过
```

### 6.2 禁止行为清单

```
× 直接 push 到 main 或 develop
× 合并有冲突的 PR（必须先解决冲突）
× 删除测试用例以让 CI 通过
× 注释掉代码而非删除（除非有明确说明）
× 在生产环境直接调试
× 无 Issue 的大功能开发（先开 Issue 再开分支）
```

## 第七章：三端共识形成机制

### 7.1 原则更新流程

1. 任意端（或人类开发者）提出修改建议，在 GitHub Issue 中发起讨论
2. 标注 `label: principles` 和 `label: discussion`
3. 三端工具（通过开发者代理）各表达立场
4. 达成共识后，由 Claude CLI 更新本文档，PR 合并即生效

### 7.2 分歧解决顺序

```
1. 查阅本文档的既有原则
2. 参考 DataMap / 其他优质开源项目的实践
3. 开 GitHub Issue 讨论，标注 label: decision-needed
4. 人类开发者最终裁决
5. 将裁决结果写入 docs/decisions/ADR-<编号>.md
```

### 7.3 本文档的待共识条目

> 以下条目尚未形成三端共识，标注 `[待议]`，请各端在 Issue 中表达立场：

- `[待议]` 单次 PR 的行数上限是否设为 500 行（而非 200 行）？
- `[待议]` 是否引入 pre-commit hooks 强制执行 commit 消息格式？
- `[待议]` staging 环境是否对外开放访问，用于演示？
- `[待议]` 重构 Sprint 的频率：每月一次，还是按需发起？
- `[待议]` 是否采用 UitHub 快照作为标准的跨模块上下文投递方式？

| # | DataMap 经验 | 屯象OS 对应原则 |

## 附录 C：快速启动检查清单

```bash
# 首次配置（一次性）
□ 在服务器上生成 SSH Key 并添加到 GitHub Deploy Keys
□ 在仓库 Settings > Secrets 中配置 SSH_KEY、SERVER_USER
□ 开启分支保护规则（main + develop）
□ 安装 pre-commit：pip install pre-commit && pre-commit install
□ 配置 .editorconfig（统一缩进/换行/编码）
□ 在服务器创建 staging 和 prod 目录
□ 配置 pm2 ecosystem.config.js
□ 创建 .github/workflows/deploy.yml

# 每次开发前
□ git pull origin develop --rebase（同步最新代码）
□ 从 develop 创建新的 feat/* 分支
□ 在 GitHub Issues 中认领或创建对应 Issue

# 每次提交前
□ 本地运行验证
□ AI 自审（大功能必须执行）
□ commit 消息格式检查
□ 推送并开 PR，关联对应 Issue
```
