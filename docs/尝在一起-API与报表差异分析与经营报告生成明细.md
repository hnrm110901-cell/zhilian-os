# 尝在一起 · API 与报表差异分析与经营报告生成明细

**品牌**：尝在一起  
**目的**：① 倒推 API 拉取与品智后台报表导出数据差异原因；② 基于麦肯锡式经营分析查漏补缺；③ 输出汇报文件生成明细供与品智报表逐项比对。  
**数据前提**：所有数据均来自尝在一起数据库（API 与报表同源），差异来自口径、范围、时点或汇总方式。

**重要**：经营报告中的月度数据（如 2025 年 12 月、2026 年 1 月实收金额、订单笔数）**必须使用从 API 拉取的真实数据或品智报表导出值，不得使用推算值**。  
- **从 API 拉取真实数据**：在已配置品智 API 的环境运行  
  `python3 scripts/fetch_pinzhi_monthly_data.py --month 2025-12`  
  或 `--month 2026-01`，脚本会输出该月总营业实收(元)、订单笔数、客单价，可直接填入报告。  
- 脚本路径：`zhilian-os/scripts/fetch_pinzhi_monthly_data.py`。

## 一、API 与品智报表导出差异原因倒推

以下为「同源数据却出现不一致」时的**可能根因**，用于与品智导出报表逐项核对、定位差异。

### 1.1 时间与日期口径

| 序号 | 差异原因 | 说明 | 建议核对 |
|------|----------|------|----------|
| 1 | **营业日 vs 自然日** | API 的 `businessDate` 可能按「营业日」（如当日凌晨至次日凌晨）统计；品智报表可能按自然日 0:00–24:00 或按「结账日」统计。跨日结账单会归属不同日期。 | 品智报表是否选「营业日」/「自然日」/「结账日」；选同一口径后对比单日合计。 |
| 2 | **时区与结账时点** | 订单归属日若按 `openTime` / `payTime` 不同，或服务器时区与门店时区不一致，会导致同一单在不同系统落在不同日。 | 取 1–2 天样本，按 billId 对比 API 订单的 payTime 与报表中该单归属日。 |
| 3 | **月份边界** | 1 月 31 日 23:xx 结账可能被 API 算在 1 月、报表算在 2 月（或反之），导致月度总营收/笔数差一单。 | 对比 1 月最后一天与 2 月第一天的日汇总，看边界单是否一致。 |

### 1.2 金额口径

| 序号 | 差异原因 | 说明 | 建议核对 |
|------|----------|------|----------|
| 4 | **金额字段不一致** | API 订单有 `dishPriceTotal`（原价）、`specialOfferPrice`（优惠）、`realPrice`（实收）。报表「总营业实收」若 = 实收，则应对应 `realPrice` 汇总；若含抹零/舍入规则不同会差几分到几元。 | 确认品智报表「总营业实收」对应字段（realPrice / 应收减优惠 / 其他）；用同一字段在 API 侧汇总对比。 |
| 5 | **单位：元 vs 分** | 品智 API 部分接口金额为**分**（整数），若按「元」汇总会差 100 倍；或四舍五入顺序不同（先按单舍入再加总 vs 先加总再舍入）导致尾差。 | 确认 API 文档与适配器是否统一 ÷100；对比单日总金额尾数。 |
| 6 | **退款与冲红** | 退款/部分退/冲红单是否包含在「总营业实收」：报表可能为「实收净额（已扣退款）」而 API 若只筛 billStatus=已结账可能含退款前金额，或退款单单独表未扣减。 | 看品智报表是否有「退款」/「净营收」；API 侧是否扣减退款或排除退单。 |
| 7 | **挂账/未收** | 挂账、签单等是否计入「总营业实收」：报表可能只计已到账，API 的 queryOgnDailyBizData 可能含挂账。 | 确认双方是否都只计「已到账实收」或都含挂账。 |

### 1.3 范围与汇总方式

| 序号 | 差异原因 | 说明 | 建议核对 |
|------|----------|------|----------|
| 8 | **门店/组织范围** | API 不传 `ognid` 时是否等于「全部门店」；报表导出时勾选的门店/品牌是否与 API 一致（如是否含试营业、已关店）。 | 列出 API 侧参与汇总的 ognid 列表，与报表「导出范围」逐店一致。 |
| 9 | **多主体/多账套** | 尝在一起若有多法人/多账套，报表可能按「主体」导出，API 按 token 对应主体，需同一主体对比。 | 确认 token 与报表选择的是同一主体/账套。 |
| 10 | **sum 与 list 不一致** | queryOgnDailyBizData 的 `sum` 与 `list` 加总可能因品智内部逻辑不完全一致（如 sum 含补录、list 仅流水），导致用 list 逐单加总 ≠ sum。 | 用 API 返回的 sum 总营收与「list 逐单 realPrice 加总」对比；以品智报表口径为准时，优先采用与报表一致的来源（sum 或 list）。 |

### 1.4 分页与去重

| 序号 | 差异原因 | 说明 | 建议核对 |
|------|----------|------|----------|
| 11 | **订单分页遗漏** | queryOrderListV2 分页拉取时，若未遍历全部页码，只汇总了前 N 页，则订单笔数与总金额会少于报表。 | 月度拉取时循环 pageIndex 直至无数据，再汇总笔数与金额，与报表对比。 |
| 12 | **重复单** | 同一 billId 跨天或跨门店被统计两次；或 API 去重逻辑与报表不同。 | 按 billId 去重后再汇总；对比去重后笔数。 |
| 13 | **billStatus 过滤** | 仅「已结账」才计入营收；若 API 与报表对「已结账」定义不同（如含/不含部分结账、并单），会差笔数与金额。 | 确认双方对 billStatus=1（已结账）及并单、拆单规则是否一致。 |

### 1.5 汇总层级与字段映射

| 序号 | 差异原因 | 说明 | 建议核对 |
|------|----------|------|----------|
| 14 | **用错接口汇总** | 总营收应用 queryOgnDailyBizData（sum）或 queryStoreSummaryList 各店加总，而非仅 queryOrderListV2 逐单加总（易受分页影响）。 | 以 queryOgnDailyBizData 的 sum 或 queryStoreSummaryList 加总为「API 侧总营收」与报表对比。 |
| 15 | **餐段/渠道维度** | 餐段、堂食/外卖/外带若从 list 或 sum 子字段汇总，需与品智报表的维度定义一致（如餐段边界时间、渠道来源字段）。 | 导出品智「按餐段」「按渠道」报表，与 API 同维度对比。 |
| 16 | **支付方式** | payment.do / payType.do 为支付方式主数据；按支付方式汇总金额需从订单 paymentList 或流水汇总，与报表「按支付方式实收」对比。 | 确认报表「按支付方式」是订单维度还是到账流水维度；API 侧用同一逻辑。 |

### 1.6 建议的逐项比对动作（与品智导出报表）

1. **总营业实收（月度）**  
   - API：按日循环 `queryOgnDailyBizData(businessDate=每日, ognid=不传)`，取每日 `sum` 中对应「总实收」字段加总；或 `queryStoreSummaryList` 按日取各店营收加总再按日加总。  
   - 报表：品智 2026 年 1 月「总营业实收」导出值（您已提供 3,253,861.54 元）。  
   - 对比：两者应一致；若不一致，按 1.1–1.5 逐条排查（日期口径、金额字段、门店范围、是否含退款/挂账）。

2. **订单笔数（月度）**  
   - API：`queryOrderListV2(beginDate=2026-01-01, endDate=2026-01-31)` 全部分页拉取，按 billId 去重，只计 billStatus=已结账。  
   - 报表：品智 1 月「订单笔数」或「账单数」。  
   - 对比：一致则口径统一；不一致则查分页是否取全、是否去重、状态筛选是否一致。

3. **按门店营收（月度）**  
   - API：按日循环 `queryStoreSummaryList(businessDate=每日)`，按 ognid/门店汇总各店当月合计；或按日循环 `queryOrderSummary(ognid=店, businessDate=每日)` 再按店加总。  
   - 报表：品智「按门店」月度营收。  
   - 对比：各店金额一致则门店范围与汇总逻辑对齐。

4. **按餐段 / 按渠道**  
   - API：从 queryOgnDailyBizData 的 sum 中 `consumeAmount_{餐段}_{类型}` 或 list 中对应字段汇总。  
   - 报表：品智「按餐段」「按渠道」导出。  
   - 对比：维度定义一致后，差异多来自字段映射或边界规则。

## 二、麦肯锡式经营分析还缺哪些经营参数

麦肯锡式报告通常具备：**结构化问题树 → 事实与数据 → 对标/基准 → 根因 → 方案与建议 → 执行与跟踪**。以下按「当前报告已有」与「仍缺」列出经营参数，便于补全。

### 2.1 当前报告已覆盖

| 维度 | 参数 | 数据来源（API/报表） |
|------|------|----------------------|
| 营收 | 总营业实收、同比环比 | 品智报表导出（总营业实收） |
| 营收结构 | 按门店、按餐段、按渠道（堂食/外卖/外带） | 品智 queryOgnDailyBizData / queryStoreSummaryList；报表可校验 |
| 订单与单价 | 订单笔数、客单价、按门店订单与客单价 | queryOrderListV2 全页+去重；或报表 |
| 成本与毛利 | 营业成本、毛利率、成本率 | 需供应链/仓管或 FCT；当前为估算或占位 |
| 支付结构 | 微信/支付宝/会员/现金占比 | 订单 paymentList 或 payType + 流水汇总；报表可校验 |

## 三、汇报文件生成明细（与品智报表逐项比对用）

以下为「经营分析汇报文件」的**生成明细清单**：每一行对应报告中的一块内容、其数据来源（API 接口与字段或报表导出项）、以及是否需与品智报表核对。便于您按表从品智导出对应报表做逐一比对，并标注差异原因。

### 3.1 封面与概述

| 序号 | 汇报内容 | 数据项 | API/数据来源 | 品智报表比对项 | 备注 |
|------|----------|--------|--------------|----------------|------|
| 1 | 报告期、品牌 | 报告期、品牌名称 | 固定：2026-01-01～01-31，尝在一起 | — | 无需比对 |
| 2 | 总营业实收（元/万元） | 月度总营业实收 | **以品智报表导出为准**（您已提供 3,253,861.54 元）；API 校验用 queryOgnDailyBizData 按日 sum 加总 或 queryStoreSummaryList 按日各店加总 | 品智「总营业实收」或「实收合计」 | 差异见 一、1.1–1.5 |
| 3 | 订单笔数 | 月度已结账订单数 | queryOrderListV2 全部分页、billId 去重、billStatus=已结账 | 品智「订单笔数」/「账单数」 | 分页与去重需一致 |
| 4 | 客单价（元） | 总营业实收 ÷ 订单笔数 | 由 2、3 计算 | 同上 | — |
| 5 | 综合毛利率 | 毛利 ÷ 营收 | 供应链/FCT 或估算；成本来自领用/采购 | 若有「成本/毛利」报表可核对 | 当前可能为估算 |

### 3.2 营收分析

| 序号 | 汇报内容 | 数据项 | API/数据来源 | 品智报表比对项 | 备注 |
|------|----------|--------|--------------|----------------|------|
| 6 | 总体营收同比环比 | 本月、上月、去年同月金额与增幅 | 本月用 2；上月/去年同月需 API 或历史报表 | 品智多期「总营业实收」导出 | 口径一致才能比 |
| 7 | 按门店营收 | 各门店月度营收、占比、同比环比 | queryStoreSummaryList 按日汇总按店加总；或 queryOrderSummary 按店按日再加总 | 品智「按门店」营收报表 | 门店列表与范围一致 |
| 8 | 按餐段营收 | 早/午/下午茶/晚/夜宵金额与占比 | queryOgnDailyBizData 的 sum 中 consumeAmount_{餐段}_{类型} 或 list 按餐段汇总 | 品智「按餐段」营收报表 | 餐段定义与边界一致 |
| 9 | 按渠道营收 | 堂食/外卖/外带金额与占比 | 同上，按类型（1堂食 2外卖 3外带）汇总 | 品智「按渠道」或「堂食/外卖/外带」报表 | 渠道来源字段一致 |

|------|----------|--------|--------------|----------------|------|

## 四、建议的下一步动作

1. **对总营收与笔数做一次标准口径对齐**  
   - 用 API：queryOgnDailyBizData 按日（2026-01-01～01-31）取 sum 总实收加总；queryOrderListV2 全页、去重、已结账得笔数。  
   - 与品智导出「2026 年 1 月总营业实收」「订单笔数」对比；若有差，按 一、1.1–1.5 逐项勾选原因并记录。

2. **从品智导出以下报表做明细比对**（若有）  
   - 按日营收/订单（或日报）  
   - 按门店营收  
   - 按餐段、按渠道营收  
   - 按支付方式实收  
   - 按菜类/品类销售（若有）  
   与 API 同维度汇总结果逐项填到 **三、汇报文件生成明细** 的「品智报表比对项」列，并标注一致/差异及原因。

3. **补全麦肯锡式参数时优先**  
   - 高优先级：日均/逐日、周末 vs 工作日、同店同比、门店对比、品类/畅销 SKU、成本率/毛利趋势、目标达成。  
   - 数据来源：能来自品智 API 的用 二、 中「建议数据来源」；成本/人工/租金等来自 FCT 或 HR/供应链，在汇报明细中单独注明「非品智、来自 XX 系统」。

4. **汇报文件生成流程**  
   - 每月固定：先跑 API 拉取（或从屯象OS 已同步库）得到「API 侧」汇总；再导出品智当月报表；用 **三、汇报文件生成明细** 逐项比对并填「品智报表比对项」与备注；最后以「品智报表导出值为准」的项（如总营业实收）覆盖报告，其余用 API 或业务系统数据生成正文与附录，并附上 3.7 的口径与差异说明。
