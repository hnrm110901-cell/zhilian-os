# 智链OS 业财税资金一体化 — 财务部门可正常使用完整性评估

本文档从**实际连锁餐饮企业财务部门可正常开展工作**的角度，评估当前 FCT 模块的完整性与缺口，并给出补全优先级建议。

---

## 一、财务部门日常工作场景与所需能力

| 场景 | 典型动作 | 所需系统能力 |
|------|----------|--------------|
| **凭证与总账** | 查看/补录凭证、按科目查余额、期间结账、调账 | 凭证列表/详情、总账余额/明细、手工凭证录入、期间锁定/结账 |
| **业财对接** | 日结/采购自动生成凭证、差异处理 | 事件接入、凭证规则、与对账/适配器联动 |
| **主数据** | 维护门店、客商、科目、银行账户 | 主数据 CRUD、与业务系统同步 |
| **资金** | 查看流水、对账、录入收付款、内部划拨 | 资金流水、对账状态、收付款/划拨录入、银企或手工补录 |
| **税务** | 开票、收票、验真、勾选、申报取数 | 销项/进项发票 CRUD、与税控/全电对接、申报表取数 |
| **费控与备用金** | 备用金申请/冲销/还款、费用报销占位 | 备用金主档与流水、审批占位 |
| **预算** | 编制预算、报销/付款前占位与超支校验 | 预算编制、占用、超预算校验/拦截 |
| **报表与分析** | 期间汇总、多维度报表、计划 vs 实际、同比环比 | 报表 API、计划达成、趋势与对比 |
| **审批与协同** | 费用/付款审批、与 OA 或内部流程衔接 | 审批记录、状态回写、可选工作流对接 |
| **发票闭环** | 票-账-款关联、查重、验真、归档 | 发票与凭证关联、验真状态、按凭证查发票 |

## 二、当前能力与缺口对照

### 2.1 已具备且可支撑日常使用的部分

| 能力域 | 现状 | 财务可做的动作 |
|--------|------|----------------|
| **业财事件 → 凭证** | 门店日结、采购入库事件自动生成凭证；对账完成后可推送日结事件 | 依赖 POS/适配器推送事件即可产生凭证，无需手工录日结/采购凭证 |
| **凭证查询** | 凭证列表（筛选 tenant/entity/日期）、凭证详情（含分录） | 查凭证、核账、审计追溯 |
| **总账余额** | 按科目汇总已过账凭证借贷，返回余额 | 看科目余额、做试算（注：当前 API 注释为「占位」但实现为真实汇总） |
| **主数据** | 门店/客商/科目/银行账户 upsert 与列表 | 维护主数据、独立部署时从外部同步 |
| **资金流水与对账** | 资金流水列表、对账状态查询（占位数据） | 查看流水与对账状态；真实数据需从对账/银行侧写入 |
| **税务占位** | 发票列表、申报列表（占位）；发票与凭证关联、验真占位 | 维护发票与凭证关联、标记验真；真实开票/申报需对接税控/全电 |
| **费控/备用金** | 备用金主档、申请/冲销/还款流水、余额更新 | 门店备用金管理、流水留痕 |
| **预算** | 预算 upsert、占用、超预算校验（check） | 编制预算、报销/付款前校验与占用 |
| **审批占位** | 按业务单创建/查询审批记录 | 记录审批状态，与 OA/工作流扩展对接 |
| **报表** | 期间汇总、四维汇总、趋势、按门店/区域、同比环比、计划 vs 实际 | 经营分析、计划达成分析、多维度报表 |
| **年度计划** | 计划 upsert、计划 vs 实际（日/周/月/季） | 目标管理与达成跟踪 |
| **FCT Agent** | 报表查询、凭证解读、对账状态 | 自然语言查报表/凭证，适合移动端或门户集成 |

以上能力**已能支撑**：业财自动凭证、凭证与总账查询、主数据维护、备用金与预算管理、发票与凭证关联、报表与计划分析、审批留痕。财务在**有 API 调用方（前端/集成/脚本）**的前提下，可以开展日常核算、对账准备、预算控制与经营分析。

### 2.2 部分具备或需补强的部分

| 能力域 | 现状 | 缺口与影响 |
|--------|------|------------|
| **总账** | 有余额汇总，无「总账明细」（按科目+期间的借贷明细列表） | 财务查某科目某月明细需自行从凭证汇总或加 API |
| **凭证状态与过账** | 凭证有 status（draft/pending/approved/posted/rejected），无「过账」API | 若业务只写 draft，需要显式过账接口才能进总账汇总；当前规则引擎生成后状态需确认 |
| **资金** | 流水与对账为占位/被动写入，无「录入一笔收/付款」API | 无法在系统内直接补录银行流水或内部划拨，需对账/外部系统写入 |
| **税务** | 发票与凭证关联、验真占位有；无「销项开票请求」「进项采集」与税控/全电真实对接 | 开票、采集、勾选、申报取数需对接外部系统或后续迭代 |
| **预算与凭证联动** | 有预算占用与校验 API，凭证生成时未强制校验/占位 | 超预算仍可生成凭证，需业务侧在付款/报销前主动调 check/occupy |

### 2.3 缺失且影响「可正常使用」的部分

| 能力域 | 缺口 | 对财务日常工作的影响 |
|--------|------|----------------------|
| **手工/调整凭证** | 无「创建凭证」API（仅事件自动生成） | 计提、摊销、调账、更正等无法在系统内完成，只能依赖事件或线下处理 |
| **会计期间与结账** | 无期间定义、无期间锁/结账 | 无法按自然月/季锁账，易产生事后篡改风险；报表取数口径依赖「按日期」而非「按期间」 |
| **凭证审批流** | 仅有审批记录占位，凭证自身无审批状态驱动（如 draft→approved→posted） | 审批与凭证状态未打通，需业务约定或扩展 |
| **前端/工作台** | 无 FCT 专属前端页面 | 财务必须通过其他系统调用 API 或自建前端，无开箱即用操作界面 |
| **银企直连 / 流水导入** | 无银行流水自动拉取、无文件导入流水 | 资金流水依赖对账模块或手工在别处录入后同步，无法在 FCT 内完成「录入→对账」闭环 |
| **合并报表** | 未实现多主体合并、内部抵销 | 集团层面需在外部做合并或后续版本支持 |
| **税务申报取数** | 申报表为占位，未从总账/发票自动取数 | 申报仍需从外部系统或手工取数 |

## 三、结论：能否「正常使用」？

- **在「以 API 为主、有人机界面或集成方」的前提下**：  
  **可以**支撑财务的日常使用，包括：  
  - 业财自动凭证（日结/采购）  
  - 凭证与总账查询、主数据维护  
  - 备用金与预算管理、审批留痕  
  - 发票与凭证关联、报表与计划分析  

- **在「财务希望完全在智链OS 内闭环、无需其他系统」的前提下**：  
  仍有明显缺口，需补全至少：  
  - **手工/调整凭证创建与过账**  
  - **会计期间与结账（可选但建议有）**  
  - **资金收付款/划拨录入（或流水导入）**  
  - **FCT 工作台前端**（或明确支持的第三方/低代码对接方式）  

- **在「与税局/银行完全打通」层面**：  
  当前为占位与预留，开票、申报、银企直连需后续对接与实施。

## 四、建议补全优先级（便于财务真正「正常使用」）

| 优先级 | 能力 | 说明 | 状态 |
|--------|------|------|------|
| **P0** | 手工/调整凭证创建 API | 支持录入借贷分录、来源标识为 manual，并支持过账，否则调账与计提无法在系统内完成 | ✅ 已落地：`POST /vouchers` |
| **P0** | 凭证过账/状态变更 API | 明确 draft → posted 的接口与规则引擎生成凭证的默认状态，保证总账取数口径一致 | ✅ 已落地：`PATCH /vouchers/{id}/status` |
| **P1** | 总账明细 API | 按科目+主体+日期范围返回借贷明细，便于查账与审计 | ✅ 已落地：`GET /ledger/entries` |
| **P1** | 资金收付款/内部划拨录入 API | 至少支持手工录入一笔收/付/划拨并可选生成凭证，与现有流水表对接 | ✅ 已落地：`POST /cash/transactions` |
| **P1** | FCT 最小工作台或对接说明 | 凭证列表/详情、总账余额、报表、备用金/预算入口的简单前端，或文档说明如何用低代码/第三方对接现有 API | ✅ 已落地：见 [FCT 最小工作台对接说明](./fct-workbench-integration.md) |
| **P2** | 会计期间与结账 | 期间定义、期间锁定/结账，报表按期间取数 | 待排期 |
| **P2** | 凭证与审批/预算联动 | 创建或过账凭证时可选校验预算、回写审批状态 | 待排期 |
| **P3** | 合并报表、税务申报取数、银企直连 | 随客户需求与资源排期 | 待排期 |
