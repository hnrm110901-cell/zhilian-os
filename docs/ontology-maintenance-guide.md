> 版本: v1.0 | 创建日期: 2026-03-27
> 维护人: 屯象科技

## 一、背景与目标

屯象OS的核心竞争力在于Ontology层编码了创始人16年餐饮行业的经营认知。这些认知包括：
- 业务规则（折扣审批阈值、损耗预警线等）
- 因果判断（"损耗率超过X%时，80%原因来自备料过多"等）
- 行业基准线（湘菜/黔菜/粤菜等不同菜系的成本结构差异）

这些知识如果不系统维护，会随着时间失效、与实际经营脱节。本指南建立了一套知识维护机制，确保Ontology层持续反映真实的经营认知。

**核心原则：Ontology内容需要创始人确认，AI不自动修改。**

## 二、快照体系说明

### 2.1 什么是Ontology快照

Ontology快照是对单个客户知识图谱的"人类可读总结"，记录：
- 客户基础信息（门店、POS系统、成本目标等）
- 已编码的业务规则（经过验证的经营规则）
- 因果判断（创始人的行业经验编码）
- 知识图谱健康度（节点统计、系统集成状态）

### 2.2 文件组织

```
docs/ontology-snapshots/
├── template.md                    # 快照模板（新客户基于此创建）
├── czyz-changzaiyiqi.md          # 尝在一起快照
├── zqx-zuiqianxian.md            # 最黔线快照（待创建）
└── sgc-shanggongchu.md           # 尚宫厨快照（待创建）
```

命名规范: `{客户代号}-{客户拼音}.md`

### 2.3 与代码系统的关联

快照中记录的业务规则，在系统中通过以下路径被Agent使用：

| 快照部分 | 关联代码 | 说明 |
|---------|---------|------|
| 成本管理规则 | `seed_real_merchants.py` 中的 `target_*_pct` | 初始目标值在种子脚本中设定 |
| 菜品BOM | `ontology_adapter.py` → `get_dish_bom()` | Agent查询菜品配方 |
| 损耗规则 | `waste_reasoning_service.py` | 损耗五步推理引擎 |
| 因果判断 | `ontology_nl_query_service.py` | 自然语言查询图谱 |
| 门店数据 | `ontology_sync_service.py` → `sync_stores_to_graph()` | PG→Neo4j同步 |
| 行业基准 | `industry_benchmarks_seed.py` | 行业对比数据 |

## 三、快照创建流程（新客户Onboarding）

每个新客户接入时，按以下步骤创建Ontology快照：

### Step 1: 复制模板
```bash
cp docs/ontology-snapshots/template.md docs/ontology-snapshots/{代号}-{拼音}.md
```

### Step 2: 填写基础信息
从以下来源提取数据：
- `config/merchants/README.md` — brand_id、门店数
- `apps/api-gateway/scripts/seed_real_merchants.py` — 成本目标、门店清单
- POS系统对接文档 — POS类型、API配置状态

### Step 3: 标记待填写项
所有业务规则部分标记为"待创始人填写"，不编造数据。

### Step 4: 注册到consolidation脚本
编辑 `scripts/ontology-consolidation.py` 中的 `CUSTOMER_MAP`，添加新客户映射：
```python
CUSTOMER_MAP = {
    "新客户代号": ("新客户文件名.md", "新客户中文名"),
    ...
}
```

### Step 5: 创始人填写
安排与创始人的onboarding会议，逐项填写：
- 业务规则（折扣、成本、运营）
- 因果判断（行业经验编码）
- 待验证假设

## 四、月度Consolidation流程

每月第一周执行Ontology consolidation，确保知识库时效性。

### 4.1 运行检查脚本
```bash
# 检查所有客户快照状态
python scripts/ontology-consolidation.py

# 检查指定客户并输出检查清单
python scripts/ontology-consolidation.py --customer czyz --checklist

# 自定义过期天数（默认90天）
python scripts/ontology-consolidation.py --days 60
```

### 4.2 月度审查检查清单

| 序号 | 检查项 | 负责人 | 说明 |
|------|--------|--------|------|
| 1 | 业务规则变更确认 | 创始人 | 本月是否有新的业务规则需要编码? |
| 2 | 成本目标审查 | 创始人 | 食材/人力/租金/损耗目标是否需要调整? |
| 3 | 因果判断验证 | 创始人+数据 | 已验证的因果关系是否仍然成立? |
| 4 | 假设验证推进 | 数据团队 | 待验证假设是否有被本月数据验证或否定? |
| 5 | 废弃规则归档 | 创始人 | 是否有规则因业态变化不再适用? |
| 6 | 本体节点统计更新 | 技术团队 | 从Neo4j导出最新节点数量 |
| 7 | 系统集成状态 | 技术团队 | POS/会员/供应链连接是否正常? |
| 8 | 更新日志和版本号 | 执行人 | 记录本次审查变更，更新consolidation日期 |

### 4.3 过期规则处理

当consolidation脚本检测到超过90天未更新的快照时：
1. 所有已验证的业务规则标记为"待重新验证"
2. 安排与创始人的专项审查会议
3. 逐条确认规则是否仍然有效
4. 更新快照并记录到Consolidation日志

## 五、规则编码规范

### 5.1 业务规则格式
```markdown
- 规则名称: 具体阈值/条件
  - 来源: [创始人经验 | 数据验证 | 行业基准]
  - 验证日期: YYYY-MM-DD
  - 适用范围: [全品牌 | 指定门店]
```

### 5.2 因果判断格式
```markdown
1. 因果描述（一句话）
   - 原因变量: X
   - 结果变量: Y
   - 置信度: [高 | 中 | 低]
   - 验证方式: [数据回归 | 经验判断 | A/B测试]
   - 验证日期: YYYY-MM-DD
```

### 5.3 废弃规则格式
在"过期/废弃规则"表格中记录：
- 规则原文
- 废弃日期
- 废弃原因（业态变化/数据否定/客户要求等）

### 5.4 编码原则
1. **可执行性**: 规则必须足够具体，Agent可以直接基于规则做判断
2. **可溯源**: 每条规则标注来源（创始人/数据/行业基准）
3. **可验证**: 每条规则必须有对应的数据验证方式
4. **有时效**: 每条规则必须有验证日期，超过90天标记为待重新验证

## 六、与Agent系统的关联

### 6.1 Ontology四层架构回顾

| 层级 | 名称 | 本维护指南的关联 |
|------|------|-----------------|
| L1 感知层 | Perception | 快照"外部系统集成状态"跟踪数据源健康 |
| L2 本体层 | Ontology | 快照"本体层节点统计"跟踪图谱规模 |
| L3 推理层 | Reasoning | 快照"因果判断"和"业务规则"被推理引擎使用 |
| L4 行动层 | Action | 快照"运营规则"决定Agent何时触发行动 |

### 6.2 KnowledgeAwareAgent (ontology_adapter.py)

`KnowledgeAwareAgent` 是所有本体感知Agent的基类，提供：
- `query_ontology()` — Cypher查询Neo4j
- `get_dish_bom()` — 获取菜品BOM配方
- `get_waste_events()` — 获取损耗事件
- `explain_reasoning()` — 推理证据链(XAI可解释性)
- `get_store_knowledge_summary()` — 门店知识图谱摘要

快照中的业务规则为这些查询提供判断阈值。例如：
- 损耗目标占比(3%) → `get_waste_events()` 结果的异常判定基准
- 食材成本目标(35%) → BOM偏差分析的对比基线

### 6.3 未来演进

当Ontology快照积累足够数据后，可以：
1. **自动化节点统计**: 从Neo4j定期导出统计数据填充快照第六部分
2. **规则生效检测**: Agent执行中检测规则触发频次，辅助判断规则有效性
3. **跨客户知识迁移**: 在客户授权下，将A客户验证的规则推荐给同业态B客户

## 七、FAQ

**Q: 创始人不在时如何处理consolidation?**
A: 跳过"业务规则变更确认"和"因果判断验证"步骤，仅执行技术层面的检查（节点统计、系统集成状态）。业务层面的审查推迟到创始人可用时。

**Q: 快照中的数据与代码中的配置不一致怎么办?**
A: 以代码中的配置（如seed_real_merchants.py）为准，更新快照使其一致。快照是对代码配置的人类可读记录，不是配置的权威来源。

**Q: 新增业务规则是否需要同步修改代码?**
A: 如果规则需要被Agent执行（如新的成本阈值），则需要同步更新代码中的对应配置。快照记录"应该是什么"，代码实现"实际执行什么"。

**Q: 如何处理不同门店有不同规则的情况?**
A: 在快照的"业务规则编码"部分标注"适用范围"，区分全品牌规则和门店特有规则。
